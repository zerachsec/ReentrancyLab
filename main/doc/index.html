Reentrancy is not just a bug — it is a consequence of trusting the call stack. When a Solidity contract sends ETH and waits for a response, it hands control to an entity that may not deserve it. This post is a surgical dissection of that trust failure.

The numbers tell one story. The code tells a much more interesting one. Let me show you what's actually happening at the EVM level when a reentrancy attack fires — and more importantly, why the naive mental model of smart contract execution leads developers straight into this trap.

Section 01

The Vulnerable Pattern

Start with the most common manifestation — an ETH withdrawal function. At first glance it looks completely reasonable. There's a balance check, a transfer, and a state cleanup. The logic seems sound.

Then you read it again, this time tracing the actual execution order:

EtherVault.sol// ❌ VULNERABLE — classic reentrancy setup
function withdraw() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0, "Nothing to withdraw");

    // ← Step 2: External call — EVM pauses HERE and gives
    //           control to msg.sender's code
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "ETH transfer failed");

    // ← Step 3: This runs AFTER the attacker's fallback returns
    //           But by then, withdraw() has already been called again!
    balances[msg.sender] = 0;
}

The developer's mental model: "I check the balance, I send the ETH, then I zero the balance."

The EVM's reality: "I check the balance, I suspend execution, I give you control, you call withdraw() again while the balance is still non-zero, you call it again, you call it again… then eventually I zero the balance once for the outermost call."

⚠ Root Cause The Solidity .call() opcode — unlike transfer() — forwards all available gas to the recipient. If the recipient is a contract, that gas is more than enough to execute a full re-entrant call back into your function before you've updated your state.
Section 02

Dissecting the Attack

Let's trace through a concrete attack. Alice deposited 5 ETH. Bob deposited 3 ETH. The vault holds 8 ETH total. The attacker deposits 1 ETH to appear legitimate — bringing the vault to 9 ETH. Here's what happens next, frame by frame:

Execution trace — reentrancy loop
1. → Attacker calls vault.withdraw() | vault.balance = 9 ETH | attacker recorded balance = 1 ETH
2. → require(1 ETH > 0) ✓ — guard passes
3. → msg.sender.call{value: 1 ETH}("") — EVM sends 1 ETH and suspends vault execution
↪ Attacker's receive() fires | vault.balance = 8 ETH | attacker balance STILL = 1 ETH
↪ Re-enters vault.withdraw() | guard passes again (balance not zeroed!)
↪ Sends 1 ETH again | vault.balance = 7 ETH | suspends again…
↪ Loop repeats… 8 ETH, 7 ETH, 6 ETH, 5 ETH, 4 ETH, 3 ETH, 2 ETH, 1 ETH
↪ vault.balance = 0 ETH — loop exits (condition fails)
4. → All nested calls unwind — balances[attacker] = 0 runs 9 times… all on an already-zero value
5. → Result: Attacker holds 9 ETH. Vault holds 0. Alice and Bob lost everything.

Notice the key insight: the state write (balances[attacker] = 0) is the last thing in the function. The EVM will faithfully execute it — but by then, it's too late. The attacker has already re-entered the function 9 times with a non-zero balance each time.

The Attacker Contract

Here's the exploit contract. It's elegantly minimal. The payload is three lines in receive():

ReentrancyAttacker.solcontract ReentrancyAttacker {
    EtherVault public immutable vault;
    address public immutable owner;

    function attack() external {
        require(msg.sender == owner, "Only owner");
        vault.withdraw(); // ← kicks off the loop
    }

    // This is the payload — triggered every time vault sends ETH
    receive() external payable {
        if (address(vault).balance >= 1 ether) {
            vault.withdraw(); // ← re-enters before balance is zeroed
        }
    }
}
💡 Why .call() and not .transfer()? The original EVM transfer opcodes — transfer() and send() — cap forwarded gas at 2,300 units. That's enough for a log emit but not enough for a storage write, making reentrancy impossible. But the ecosystem moved to .call() because transfer() became unreliable after EIP-1884 raised SLOAD costs. The community traded one risk for another.

The Lendf.Me Case: ERC-777 Reentrancy

This one is particularly instructive because it shows reentrancy can arrive via token standards, not just direct ETH sends. ERC-777 tokens define a tokensToSend hook — a callback that fires before a transfer completes. The Lendf.Me protocol didn't account for this:

Simplified Lendf.Me pattern (reconstructed)function supply(address token, uint256 amount) external {
    // ERC-777 transferFrom triggers tokensToSend hook
    // — attacker's hook calls withdraw() HERE
    // — accounting hasn't been updated yet
    token.transferFrom(msg.sender, address(this), amount);

    // ← Accounting runs AFTER the callback — too late
    accountBalance[msg.sender] += amount;
}

The lesson: any external call is a potential reentrancy surface — not just explicit ETH sends. Token transfers, oracle calls, NFT mints with callbacks, flash loan callbacks — all of them hand temporary control to external code.

Section 04

Three Defences. One Correct Pattern.

❌ Vulnerable Order (Interact before Effect)

1. Check  → require(balance > 0)
2. Interact→ msg.sender.call{...}()
3. Effect  → balances[x] = 0

// Attacker re-enters at step 2
// Balance still non-zero at step 1
// Drains vault in a loop

✅ Safe Order (CEI Pattern)

1. Check  → require(balance > 0)
2. Effect  → balances[x] = 0
3. Interact→ msg.sender.call{...}()

// Re-entry now harmless:
// balance is already 0
// guard fails immediately

Defence 1: Checks-Effects-Interactions (CEI)

This is the architectural fix. It's not a library — it's a discipline. Always perform all state changes before any external calls. If a re-entrant call comes in, the state will already be consistent and guards will fail naturally.

EtherVaultFixed.sol — CEI patternfunction withdraw() external nonReentrant {
    uint256 amount = balances[msg.sender];

    // ✅ CHECK — validate state
    require(amount > 0, "Nothing to withdraw");

    // ✅ EFFECT — update state first
    balances[msg.sender] = 0;

    // ✅ INTERACT — external call comes last
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "ETH transfer failed");

    emit Withdrawn(msg.sender, amount);
}

Now if the attacker tries to re-enter: their balance was already zeroed. The require(amount > 0) check fails. Attack over.

Defence 2: ReentrancyGuard Mutex

OpenZeppelin's ReentrancyGuard adds a boolean mutex that gets locked at the start of any nonReentrant function and unlocked at the end. A re-entrant call hits the locked mutex and reverts immediately — before even checking balances.

Using OpenZeppelin's guardimport {ReentrancyGuard} from "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract EtherVaultFixed is ReentrancyGuard {
    function withdraw() external nonReentrant {
        // If this function is re-entered, the mutex causes an
        // immediate revert — attacker's receive() gets a failed call
        ...
    }
}
⚠ Important Nuance nonReentrant only prevents re-entering functions within the same contract that also carry the modifier. Cross-contract reentrancy and read-only reentrancy (calling view functions mid-execution) can bypass it. CEI is the architectural foundation — the guard is the safety net, not the solution.

Defence 3: Pull-over-Push Payments

Instead of the contract pushing ETH to users on withdrawal, let users pull their own ETH. The contract stores withdrawal credits and users claim them separately. This isolates any callback risk to the individual user, not the whole vault:

Pull payment pattern// Instead of: msg.sender.call{value: amount}("")
// Do this:

mapping(address => uint256) public pendingWithdrawals;

function initiateWithdraw() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0);
    balances[msg.sender] = 0;
    pendingWithdrawals[msg.sender] += amount; // credit, don't send
}

function claimWithdrawal() external nonReentrant {
    uint256 amount = pendingWithdrawals[msg.sender];
    require(amount > 0);
    pendingWithdrawals[msg.sender] = 0;
    (bool ok, ) = msg.sender.call{value: amount}("");
    require(ok);
}

The Fuzz Test: Proving the Invariant

The fuzz test is my favorite part of the suite. It runs 256 iterations with random deposit amounts and asserts one invariant: whatever you deposit, you always get back exactly that amount and nothing more. The fixed vault passes all 256 runs. The vulnerable vault — if you wire it up — fails the first time an attacker contract is simulated.

Fuzz test — deposit/withdraw invariantfunction test_fuzz_depositWithdrawInvariant(uint96 amount) public {
    vm.assume(amount > 0 && amount <= 5 ether);

    address user = makeAddr("fuzzer");
    vm.deal(user, uint256(amount));

    vm.startPrank(user);
    fixed_.deposit{value: amount}();
    uint256 before = user.balance;
    fixed_.withdraw();
    assertEq(user.balance, before + amount); // exact amount, always
    vm.stopPrank();
}
Closing

What to Take Away

Reentrancy persists not because developers are careless, but because the mental model of smart contract execution is subtly wrong. We think of functions as atomic units. The EVM does not agree. Every external call is a yield point — a moment where another agent gets control.

After auditing contracts, I look for reentrancy anywhere I see:

🔍 Audit Checklist — Reentrancy Risk Signals
  • → .call() with arbitrary gas forwarding
  • → ERC-777 / ERC-1155 token transfers (callback hooks)
  • → Flash loan callbacks where state is not settled before the callback
  • → NFT mint callbacks (ERC-721 onERC721Received)
  • → Any function that reads-then-writes across an external call
  • → Cross-contract calls where the callee can call back into the caller
  • → view functions read mid-reentrancy (read-only reentrancy)

The fix is simple to state and requires discipline to apply consistently: Checks → Effects → Interactions. Never let an external call touch code where your state is still stale. Add nonReentrant as a belt-and-suspenders guard for any function that moves funds.

The DAO lost 60 million dollars to this pattern in 2016. New contracts still get deployed with this exact bug today. The gap is not knowledge — it's attention.
